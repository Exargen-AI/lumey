# Testing Guide

> This guide describes how to run and add tests today.
> The long-term plan lives at `.gstack/qa-reports/BASELINE_HARDENING_PLAN.md`.
> Right now we're at **Phase 0** of that plan: foundation in place, real coverage starts in Phase 2.

## Quick commands

| What I want to do | Command |
|---|---|
| Run all unit + component tests (both packages) | `npm test` |
| Same with coverage report | `npm run test:unit:coverage` |
| Watch tests in the backend during dev | `npm run test:watch --workspace=backend` |
| Watch tests in the frontend during dev | `npm run test:watch --workspace=frontend` |
| Run the Playwright smoke spec (needs dev server on :5174) | `npm run test:e2e` |
| Run typecheck + lint + unit + e2e (full local gate) | `npm run test:all` |
| Just typecheck | `npm run typecheck` |
| Lint (errors fail; warnings shown but allowed) | `npm run lint` |
| Lint strict (Phase 1.5 — fail on any warning) | `npm run lint:strict` |
| Auto-fix the lint issues that can be auto-fixed | `npm run lint:fix` |
| Format the whole tree with Prettier | `npm run format` |
| Check formatting without writing | `npm run format:check` |
| Audit production deps for known vulns | `npm run audit:strict` |
| Scan for unused TypeScript exports | `npm run dead-code` |

The unit suite is fast (~3 seconds total today; will grow). The e2e suite needs a running dev server + seeded local Postgres and takes ~25 seconds.

## Test layers

### Layer 1 — Unit tests (Vitest)

**Backend:** `backend/src/**/*.test.ts`. Run with `npm test --workspace=backend`.

Cover pure functions in `services/`, `utils/`, `validators/`, `middleware/`. Use mocked or no DB.

```ts
// backend/src/utils/password.test.ts — pattern to copy
import { describe, it, expect } from 'vitest';
import { hashPassword, comparePassword } from './password';

describe('password utils', () => {
  it('hashes a password to a bcrypt string', async () => {
    const hash = await hashPassword('Admin@1234');
    expect(hash).toMatch(/^\$2[aby]\$\d{2}\$/);
  });
});
```

**Frontend:** `frontend/src/**/*.{test,spec}.{ts,tsx}`. Run with `npm test --workspace=frontend`.

Cover hooks, lib utils, presentational + interactive components. Uses jsdom + React Testing Library.

```tsx
// frontend/src/components/auth/Can.test.tsx — pattern to copy
import { render, screen } from '@testing-library/react';
import { Can } from './Can';
import { useAuthStore } from '@/stores/authStore';

beforeEach(() => {
  useAuthStore.setState({ permissions: [] });
});

it('renders fallback when permission is missing', () => {
  useAuthStore.setState({ permissions: ['task.view'] });
  render(<Can permission="task.edit_any" fallback={<span>locked</span>}>...</Can>);
  expect(screen.getByText('locked')).toBeInTheDocument();
});
```

### Layer 2 — Integration tests (Phase 3, deferred)

**Status: not yet implemented.** `supertest` is installed, but the Express app in `backend/src/index.ts` calls `bootstrap()` at module load time — connecting to Prisma, syncing permissions, and calling `app.listen()`. Importing it into a test would trigger all of that.

**Phase 3 will:**
1. Refactor `backend/src/index.ts` into two halves: a pure `createApp()` factory that returns the Express app, and a `bootstrap()` entry that's only called by the real server.
2. Add `backend/src/test/setup.ts` that boots a throwaway Postgres on port 5433, runs migrations, seeds, and resets between test files.
3. Write contract tests for all 224 endpoints + the role × endpoint permission matrix.

Until then, treat integration coverage as the responsibility of the Playwright E2E layer.

### Layer 3 — End-to-end (Playwright)

`frontend/tests/e2e/*.spec.ts`. Run with `npm run test:e2e` or just `npm test --workspace=frontend && npm run test:e2e`.

The smoke spec covers login + per-role landing + a few critical interactions. Today 16 specs; Phase 5 expands to ~60.

**Prereqs:**
- Dev server running on `http://localhost:5174` (`npm run dev:frontend`).
- Backend running on `http://localhost:3002` (`npm run dev:backend`).
- Local Postgres on `:5434` with seed loaded (`npm run db:seed --workspace=backend`).
- Playwright browsers installed (`npx playwright install chromium`).

## Coverage

Coverage reports are generated by Vitest's v8 provider. Each run writes `coverage/lcov.info` + `coverage/index.html` per package. The CI workflow uploads both as artifacts.

**Current thresholds: 0% (intentional).** Phase 2 raises backend service coverage to 85% (95% for auth/RBAC). Phase 4 raises frontend coverage to 70%. Once a target is set in `vitest.config.ts`, CI fails any PR that drops below it.

## CI

Two workflows run on every PR:

| Workflow | What it runs | Today |
|---|---|---|
| `.github/workflows/ci.yml` | typecheck + unit (with coverage) + build | new in Phase 0 |
| `.github/workflows/e2e.yml` | Playwright smoke spec against fresh Postgres | already existed |

Each job is independent; failures in one don't cancel the others.

## Adding a new test

1. **Backend service:** create `backend/src/services/{name}.service.test.ts` alongside the service file. Use the `password.test.ts` pattern. Aim for the coverage tier defined in the plan (95% for auth/RBAC, 90% for data integrity, 85% for feature surface, 70% for integrations).
2. **Frontend component:** create `{Name}.test.tsx` alongside the component. Use the `Can.test.tsx` pattern. RTL queries (`getByRole`, `getByText`, `getByLabelText`) preferred over CSS selectors.
3. **Frontend hook:** create `{name}.test.ts` alongside the hook. Use `renderHook` from RTL. See `usePermission.test.ts`.
4. **E2E:** add to `frontend/tests/e2e/{role}-{flow}.spec.ts`. Login helper at `tests/e2e/helpers/auth.ts`.

## Conventions

- **Test files live next to the file they test.** Not a separate `__tests__/` dir. Easier to find, easier to delete with the file.
- **Names**: `{filename}.test.ts` or `.spec.ts` — both work.
- **No real network calls in unit/component tests.** Mock axios at the module level, or pass props.
- **Real DB calls only in Phase-3 integration tests.** Throwaway Postgres on `:5433`, isolated from the dev DB on `:5434`.
- **Don't disable lint rules in test files** without a comment explaining why. Test files are code.
- **One assertion per behavior.** A test that asserts five things explodes its blame radius when one fails.
